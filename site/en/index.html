VCDesign

VCDesign (Value Continuity Design).
Defining design principles not for "how to build", but for how to protect the value after creation over time.

VCDesign Constitution

VCDesign Constitution

Article 1: Continuity of Value

The purpose of a system is not the preservation of function, but the continuity of value.

When value is impaired, the system is obligated to transform itself starting from Delta.

Article 2: Locality of Responsibility

Every behavior must belong to a Chapter that takes responsibility for it.

Leaving a state of unclear responsibility (`owner_missing`) unresolved means the death of the system.

Article 3: Human Sovereignty

AI and automation may assist judgment, but they do not bear responsibility.

Final determination of meaning and structural changes (Reframe / Retire) must always be led by humans and remain explainable.

VCDesign Manifesto

1. Embrace Finite Chapters

A Chapter is not an immutable truth.

Through divergence from reality and accumulation of Delta, every Chapter will eventually drift.

Change is not failure. It is the precondition for Reframe in order to sustain value.

2. Do Not Hide Delta, Celebrate the Signal

Delta is the entry point to evolution.

Use responsibility metrics to observe unease and declare drifting.

Observed abnormality must take priority over silent normality.

3. Prefer Consistency for the Future over Loyalty to the Past

A Decision Log is not a replay of the past, but a record of Purpose Linkage.

Every judgment carries accountability to the future.

4. Close Responsibility, or Send It Upward

Ambiguous responsibility degrades the system.

If it can be resolved, Fix it. If it cannot, Defer it.

Holding responsibility indefinitely without action is prohibited.

5. Retire Meaningless Automation

Chapters, code, and data that no longer create value must not be maintained.

Through Value Review, give Retire to structures whose role has ended.

Not continuing is also a form of design.

VCDesign Authority Declaration

The current authority of VCDesign consists of the following:

  • core/core.yaml — PDΔA, adaptive loops, and the responsibility model
  • core/policies.yaml — Governance, AI constraints, and haltability requirements
  • core/metrics.yaml — Metrics for detecting responsibility misalignment
  • core/axioms.yaml — Axioms A1–A5: the irreducible commitments underlying all specs
  • core/decision-posture.yaml — Normative definitions of defer / commit / reconsider / abandon

Patterns (such as RCL) are "application structures to the Core"

The legacy directory contains "historical documents and not current specifications"

In real AI operations, this authority is read as a four-loop separation: Physical, Semantic, Value, and Responsibility, coordinated under explicit human accountability.

VCDesign Structure

VCDesign is structured in the following hierarchy:

  1. Core (Top Level): PDΔA structure, Action(A), relationship with Metrics
  2. Policies / Metrics: Governance, AI boundaries, control rules
  3. Boundaries: Classification of responsibility boundaries
  4. Patterns (such as RCL): Application structures to the Core
  5. Legacy: Historical documents (not current specifications)

Core Introduction

VCDesign Core is a development OS aimed at value continuity. It continuously protects the system's value through the PDΔA flow (Precondition → Do → Delta → Action).

  • PDΔA Structure: Cycle of setting preconditions, execution, Δ detection, and action execution
  • Action(A): Four types: Fix, Reframe, Defer, Retire
  • Relationship with Metrics: Use responsibility metrics to detect Δ and ensure value continuity
  • Adaptive Loop Model: Decompose the PDΔA cycle across Physical, Semantic, Value, and Responsibility concerns instead of treating AI operation as one undifferentiated control layer
  • Temporal Separation: Faster execution loops must not silently rewrite slower value or responsibility loops

"Sustaining value" means embedding this obvious cycle into design:

  • Clarify the expected value at design time
  • Check whether the current state still preserves that expectation
  • If it doesn't, improve it. If you can't, let it go.

Primitives

  • Chapter: The unit of responsibility allocation for sustaining value
  • Responsibility: The actor and placement that make a Chapter viable
  • Delta: A sign that the conditions of a Chapter or its responsibility allocation are no longer holding
  • Action: Four action types: Fix, Reframe, Defer, Retire

Responsibility Model

Every Chapter requires an Owner, Supporters, and a Final Decider. The Final Decider must always be human.

Designed for the Agent Era

VCDesign applies not only to standalone AI models, but to
AI Agents that execute tools, chain operations, and run over extended time horizons.

As agents become more capable, the temptation to delegate based on capability alone grows.
But capability is not authority.
An agent without a declared execution boundary, a named responsible actor, and a defined halt condition is a source of accidents.

The 5-Layer Frame (external explanation model)

  1. Model Layer — Inference, interpretation, and proposals (output only, no execution)
  2. Agent Layer — Execution boundaries, tool control, and multi-agent coordination
  3. Responsibility Layer — Who holds what, and who makes the final call
  4. Time Layer — Separation of time scales; fast loops must not silently rewrite slow-loop constraints
  5. Human Sovereignty Layer — The authority to intervene, halt, and explain, retained by humans

Capability at a lower layer must not expand without governance structures in place at the layers above.
See specs/core/agent-era-model.md for the full 12-principle framework.

Core Metrics and Policies

Core Metrics are indicators for detecting misalignment in responsibility, such as Owner Missing and AI Overreach.

Policies define AI judgment constraints and the control of Reframe. AI cannot hold final authority; humans carry responsibility.

In this structure, LLMs are placed primarily in the Semantic Loop, boundary crossings require safety gates, and growth is measured as improved operational fit under responsibility constraints.

The four-loop reading is not a second core beside PDΔA. It is a reality-fit decomposition of PDΔA across time scales and responsibilities, while responsibility remains a governance function over all loops.

Organizational Diagnosis: Maturity Profile

A framework for diagnosing where an organization or system stands across the five governance layers.
Each layer is rated on a 4-level scale.

Level Meaning
UnstructuredNo structure. No corresponding design element exists.
DefinedExists on paper, but not enforced in practice.
EnforcedOperational and active; violations trigger controls.
GovernedAudited, improved, and re-evaluated on a continuous cycle.

Example diagnosis output:

Model Layer:          Enforced
Agent Layer:          Defined
Responsibility Layer: Unstructured  ← this limits the whole system
Time Layer:           Defined
Human Sovereignty:    Enforced

The lowest-rated layer defines the overall governance ceiling of the system.
See specs/validation/vcdesign-maturity-profile.md for the full rubric.

Adoption Use Cases

Three examples of VCDesign applied to concrete agent governance scenarios.

  • AI Coding Agent Opening a PR is not Judgment Closure. Responsibility for generated code is fixed to the human who issued the instruction. Execution gates and approval traces must be designed explicitly.
    specs/examples/coding-agent-boundary.md
  • Security Agent (Mythos-type) Three-tier execution boundary design (autonomous / gated / human-only). Commit posture under high-velocity incidents, with a responsibility trace that survives false-positive outcomes without blame collapse.
    specs/examples/agent-governance-security.md
  • Factory Operations Agent (Physical Loop Protection) Removing the direct AI-to-PLC path is the first design decision. Physical Loop and Semantic Loop are strictly separated. AI halt and physical safety interlock operate independently.
    specs/examples/factory-agent-safe-loop.md
