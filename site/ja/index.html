VCDesign

(Value Continuity Design)

VCDesign（Value Continuity Design）。
「どう作るか」ではなく、作られた後の価値を、
時間の中でどう守り抜くかの設計原則を定義します。

VCDesign 憲法

VCDesign 憲法 (Constitution)

第1条：価値の継続 (Continuity of Value)

システムの目的は機能の維持ではなく、価値の継続である。

価値が損なわれた時、システムはΔ（ズレ）を起点として自らを変容させる義務を負う。

第2条：責任の所在 (Locality of Responsibility)

すべての振る舞いには、それを引き受ける「章（Chapter）」が存在しなければならない。

責任が不明瞭な状態（owner_missing）を放置することは、システムの終焉を意味する。

第3条：人間の主権 (Human Sovereignty)

AIと自動化は判断を助けるが、責任を持たない。

最終的な意味の確定および構造変更（Reframe / Retire）は、常に人間が主導し、説明可能でなければならない。

VCDesign マニフェスト (Manifesto)

1. 章の有限性を愛せ (Embrace Finite Chapters)

章（Chapter）は不変の真理ではない。

現実との乖離やΔの蓄積により、章は必ず漂流（Drifting）する。

変化は失敗ではなく、価値継続のための遷移（Reframe）の前提である。

2. Δを隠すな、徴候を祝え (Celebrate Delta Signals)

Δ（ズレ）は進化の入口である。

責任メトリクスを用いて違和感を観測し、drifting を宣言せよ。

沈黙する正常よりも、観測された異常を優先せよ。

3. 過去への忠実さより、未来への一貫性を (Consistency over Legacy)

判断ログ（Decision Log）は過去の再現ではなく、「価値への接続（Purpose Linkage）」を記録するものである。

すべての判断は未来への説明責任を持つ。

4. 責任を閉じよ、さもなくば上位へ送れ (Close or Defer)

曖昧な責任はシステムを劣化させる。

解決できるならFixせよ。できないならDeferせよ。

責任を持ったまま放置することを禁止する。

5. 意味のない自動化を引退させよ (Retire the Meaningless)

価値を生まない章、コード、データは維持してはならない。

Value Reviewを通じて、役割を終えた構造にはRetireを与えよ。

継続しないこともまた、設計である。

VCDesign Authority 宣言

VCDesignの現行authorityは以下で構成される

  • core/core.yaml — PDΔA・適応ループ・責任モデルの基底
  • core/policies.yaml — ガバナンス・AI制約・停止可能性の規範
  • core/metrics.yaml — 責任逸脱の観測指標群
  • core/axioms.yaml — A1〜A5：すべての仕様の基底となる公理
  • core/decision-posture.yaml — defer / commit / reconsider / abandon の規範的定義

patterns（RCL等）は「Coreへの適用構造」

legacy配下は「歴史的ドキュメントであり現行仕様ではない」

現実のAI運用では、このauthorityは Physical / Semantic / Value / Responsibility の4ループ分離として読まれ、 その統治は常に人間の責任下に置かれます。

VCDesign の構造

VCDesign は以下の階層構造で構成されます：

  1. Core（最上位）: PDΔA構造、Action(A)、Metricsとの関係
  2. Policies / Metrics: ガバナンス、AI境界、制御ルール
  3. Boundaries: 責任境界の分類
  4. Patterns（RCL等）: Coreへの適用構造
  5. Legacy: 歴史的ドキュメント（現行仕様ではない）

Core紹介

VCDesign Coreは、価値継続を目的とした開発OSです。PDΔAフロー（Precondition → Do → Delta → Action）でシステムの価値を継続的に守ります。

  • PDΔA構造: 事前条件設定、実行、Δ検知、アクション実行のサイクル
  • Action(A): Fix, Reframe, Defer, Retire の4タイプ
  • Metricsとの関係: 責任メトリクスを用いてΔを検知し、価値継続を確保
  • Adaptive Loop Model: AI運用を単一制御層に潰さず、PDΔAを Physical / Semantic / Value / Responsibility の観点へ分解して扱う
  • 時間スケール分離: 高速な実行ループが、低速な価値判断や責任判断を書き換えないようにする

「価値を継続させる」とは、

  • 設計時の期待（価値）を明確にする
  • 今の状態がその期待を保てているか確認する
  • 保てていなければ、改善する。できなければ諦める
    • という「当たり前のサイクル」を設計に織り込む事を指します。

プリミティブ

  • Chapter: 価値を継続させるための責任配置単位
  • Responsibility: 章を成立させる主体およびその配置
  • Delta: 章の成立条件または責任配置の維持可能性が崩れた兆候
  • Action: Fix, Reframe, Defer, Retire の4タイプ

責任モデル

各章にはOwner, Supporters, Final Deciderが必要です。Final Deciderは常に人間でなければなりません。

Agent Era への対応

VCDesign は、モデル単体の AI 時代だけでなく、
ツール実行・連鎖操作・長時間動作を行う AI Agent の時代にも適用できます。

Agent が高性能になるほど、「能力があるから任せる」という判断が増えます。
しかし 能力は権限ではありません。
実行境界・責任者・停止条件を設計しない Agent は事故源になります。

5層構造（外部説明フレーム）

  1. Model Layer — 推論・解釈・提案（出力のみ、実行なし）
  2. Agent Layer — 実行境界・ツール制御・Multi-Agent 協調
  3. Responsibility Layer — 誰が何を保持し、誰が最終判断するか
  4. Time Layer — 時間スケールの分離と、速いループが遅いループを上書きしないための構造
  5. Human Sovereignty Layer — 人間が最終的に介入・停止・説明できる主権

下位レイヤーの能力は、上位レイヤーの統治構造なしに拡大してはなりません。
詳細は specs/core/agent-era-model.md を参照してください。

Core Metrics と Policies

Core Metricsは責任のズレを検知するための指標群です。例: Owner Missing, AI Overreach。

PoliciesはAIの判断制約やReframeの制御を定義します。AIは最終判断を持てず、人間が責任を担います。

その上で、LLMは主としてSemantic Loopに限定され、越境には安全ゲートと停止経路が必要であり、 成長は責任制約下での運用整合の改善として測られます。

この4ループ読解は PDΔA と並ぶ第二の中核ではなく、PDΔA を時間スケールと責任配置に沿って現実運用へ展開した説明です。 Responsibility は計算ループではなく、全ループへ介入できる統治機能として扱われます。

導入診断：Maturity Profile

組織・システムが AI ガバナンスの各層においてどの段階にあるかを診断するフレームワークです。
5層それぞれを 4段階 で評価します。

Level 意味
Unstructured構造なし。設計上の対応物が存在しない
Defined文書上は存在するが、実運用では強制されていない
Enforced実運用で機能しており、違反時に制御が発動する
Governed監査・改善・再評価まで回っており、継続的に維持される

診断例：

Model Layer:          Enforced
Agent Layer:          Defined
Responsibility Layer: Unstructured  ← ここが全体の限界
Time Layer:           Defined
Human Sovereignty:    Enforced

最も低いレベルの層が、システム全体のガバナンス限界を決めます。
詳細は specs/validation/vcdesign-maturity-profile.md を参照してください。

導入ユースケース

VCDesign を具体的な Agent ガバナンスに適用した事例です。

  • AI コーディング Agent PR オープン ≠ Judgment Closure。コード生成物の責任者を「指示を出した人間」に固定し、 実行ゲートと承認トレースを設計する。
    specs/examples/coding-agent-boundary.md
  • セキュリティ Agent（Mythos 型） 影響水準3階層（自律可 / ゲート必要 / 人間決定必須）による実行境界設計。 高速インシデント対応での commit ポスチャーと誤検知時の責任トレース。
    specs/examples/agent-governance-security.md
  • 工場運用 Agent（物理ループ保護） AI から PLC への直接パス禁止が最優先設計決定。 Physical Loop と Semantic Loop を厳格に分離し、 AI 停止と物理安全機構を独立して設計する。
    specs/examples/factory-agent-safe-loop.md

2026年時点での実装(value-review,VC-AD,TDDD)

  • AIを最大限活かすための境界定義（設計）：VC-AD AI時代に見落とされがちな境界の地図を定義し、Responsibilityの配置を支援
  • Reality Fit / Adaptive Loop Model Core・Policies・Metrics が現実のAI運用ループと停止可能な境界設計にどう接続するかを説明
  • 対話AIとコード自動生成AI活用（開発）：TDDD 生成AI時代の実務的な開発運転方法の記録。Delta観測とAction選択を実践
