VCDesign

(Value Continuity Design)

VCDesign（Value Continuity Design）。
「どう作るか」ではなく、作られた後の価値を、
時間の中でどう守り抜くかの設計原則を定義します。

VCDesign 憲法

VCDesign 憲法 (Constitution)

第1条：価値の継続 (Continuity of Value)

システムの目的は機能の維持ではなく、価値の継続である。

価値が損なわれた時、システムはΔ（ズレ）を起点として自らを変容させる義務を負う。

第2条：責任の所在 (Locality of Responsibility)

すべての振る舞いには、それを引き受ける「章（Chapter）」が存在しなければならない。

責任が不明瞭な状態（owner_missing）を放置することは、システムの終焉を意味する。

第3条：人間の主権 (Human Sovereignty)

AIと自動化は判断を助けるが、責任を持たない。

最終的な意味の確定および構造変更（Reframe / Retire）は、常に人間が主導し、説明可能でなければならない。

VCDesign マニフェスト (Manifesto)

1. 章の有限性を愛せ (Embrace Finite Chapters)

章（Chapter）は不変の真理ではない。

現実との乖離やΔの蓄積により、章は必ず漂流（Drifting）する。

変化は失敗ではなく、価値継続のための遷移（Reframe）の前提である。

2. Δを隠すな、徴候を祝え (Celebrate Delta Signals)

Δ（ズレ）は進化の入口である。

責任メトリクスを用いて違和感を観測し、drifting を宣言せよ。

沈黙する正常よりも、観測された異常を優先せよ。

3. 過去への忠実さより、未来への一貫性を (Consistency over Legacy)

判断ログ（Decision Log）は過去の再現ではなく、「価値への接続（Purpose Linkage）」を記録するものである。

すべての判断は未来への説明責任を持つ。

4. 責任を閉じよ、さもなくば上位へ送れ (Close or Defer)

曖昧な責任はシステムを劣化させる。

解決できるならFixせよ。できないならDeferせよ。

責任を持ったまま放置することを禁止する。

5. 意味のない自動化を引退させよ (Retire the Meaningless)

価値を生まない章、コード、データは維持してはならない。

Value Reviewを通じて、役割を終えた構造にはRetireを与えよ。

継続しないこともまた、設計である。

VCDesign Authority 宣言

VCDesignの現行authorityは以下で構成される

  • core/core.yaml
  • core/policies.yaml
  • core/metrics.yaml

patterns（RCL等）は「Coreへの適用構造」

legacy配下は「歴史的ドキュメントであり現行仕様ではない」

VCDesign の構造

VCDesign は以下の階層構造で構成されます：

  1. Core（最上位）: PDΔA構造、Action(A)、Metricsとの関係
  2. Policies / Metrics: ガバナンス、AI境界、制御ルール
  3. Boundaries: 責任境界の分類
  4. Patterns（RCL等）: Coreへの適用構造
  5. Legacy: 歴史的ドキュメント（現行仕様ではない）

Core紹介

VCDesign Coreは、価値継続を目的とした開発OSです。PDΔAフロー（Precondition → Do → Delta → Action）でシステムの価値を継続的に守ります。

  • PDΔA構造: 事前条件設定、実行、Δ検知、アクション実行のサイクル
  • Action(A): Fix, Reframe, Defer, Retire の4タイプ
  • Metricsとの関係: 責任メトリクスを用いてΔを検知し、価値継続を確保

「価値を継続させる」とは、

  • 設計時の期待（価値）を明確にする
  • 今の状態がその期待を保てているか確認する
  • 保てていなければ、改善する。できなければ諦める
    • という「当たり前のサイクル」を設計に織り込む事を指します。

プリミティブ

  • Chapter: 価値を継続させるための責任配置単位
  • Responsibility: 章を成立させる主体およびその配置
  • Delta: 章の成立条件または責任配置の維持可能性が崩れた兆候
  • Action: Fix, Reframe, Defer, Retire の4タイプ

責任モデル

各章にはOwner, Supporters, Final Deciderが必要です。Final Deciderは常に人間でなければなりません。

Core Metrics と Policies

Core Metricsは責任のズレを検知するための指標群です。例: Owner Missing, AI Overreach。

PoliciesはAIの判断制約やReframeの制御を定義します。AIは最終判断を持てず、人間が責任を担います。

2026年時点での実装(value-review,VC-AD,TDDD)

  • システム価値の定義（企画）：value-review Value Review Sheet を中心に、価値と境界を確定し、ChapterのPreconditionを定義
  • AIを最大限活かすための境界定義（設計）：VC-AD AI時代に見落とされがちな境界の地図を定義し、Responsibilityの配置を支援
  • 対話AIとコード自動生成AI活用（開発）：TDDD 生成AI時代の実務的な開発運転方法の記録。Delta観測とAction選択を実践
