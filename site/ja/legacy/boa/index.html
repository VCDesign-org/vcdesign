BOA Core

BOA Core は、
AI が関与するシステムにおいて、
判断・責任・停止を壊さずに扱うための構造群です。

このリポジトリは「正しい答えを出す」ためのものではありません。
「判断できない状態を、正しく止める」ために存在します。

これはフレームワークではない

BOA / RCA / RP は、一般的な意味でのレイヤードアーキテクチャやフレームワークではありません。

  • 処理を分割するための層ではない
  • 責務を整理するための役割分担でもない

これらはすべて、
判断の流れに沿って現れる「関所」です。

含まれる3つの構造

  • BOA (Boundary-Oriented Architecture) — 判断が許される境界を宣言する
  • RCA (Responsibility Closure Agent) — 判断の責任を引き受け、完了させる
  • RP (Resolution Protocol) — 判断を進める／止める／戻すことを明示的に選ぶ

重要なのは、これらが同時に存在するのではないという点です。

判断の基本フロー（要約）

  1. システムは BOA に到達する
  2. RCA が Assessment（判断引き受けの評価）を行う
  3. RCA が「引き受ける（Accepted）」とした場合のみ RP が起動する
  4. RP は必ず以下のいずれかを返す
    • Resolve（決める）
    • Reject（却下する）
    • Return（人や上流に戻す）
  5. RCA が、この判断の責任を閉じる

BOA が先に置かれる理由

BOA は最初の技術ではありません。
最初に置かれるべき境界の宣言です。

  • AI の判断がどこまで許されるのか
  • どこで止めるのか
  • 誰の責任になるのか

これが曖昧なままでは、RP も RCA も意味を持ちません。

3つの役割と問い

BOA / RCA / RP は、それぞれ異なる次元の役割を担います。
「場所・人・ルール」に似た関係です。

BOA — Boundary-Oriented Architecture

問い：「どこで判断を行うべきか？」

BOA はシステムを境界で分離する構造です。
AI が判断してよい領域と、人間が判断すべき領域を分けます。

RCA — Responsibility Closure Agent

問い：「この判断を引き受けるか？」

RCA は BOA で定義された場所で判断を行う主体です。
ACCEPTED / DENIED / UNKNOWN の意思を示します。

RP — Resolution Protocol

問い：「それは正式な解決として認められるか？」

RP は手続きの正当性を検査します。
署名・必須項目・責任主体が揃っているかを確認します。

失敗・停止・返却の扱い

BOA Core が最も避けるべき失敗は、
判断できないのに判断が進んでしまうことです。

Stop（停止）は正常系です。
境界条件を満たしていない場合、判断は開始されません。

Return（返却）は責任ある動作です。
「今は決められない」「この条件では決めるべきではない」ことを明示します。

沈黙は許されません。
何も返さない・暗黙に成功とみなすことは、責任を消します。

BOA Core は、
判断を自動化するための設計ではなく、
判断を壊さないための設計です。
