VMS

VCDesign Management Strategy

VMS は、経営上の悩みを VCDesign の Loop、Boundary、Decision に変換し、 判断の責任と価値継続を設計するための運用プロトコルです。

Purpose

VMS の目的は、経営者の意思決定における「責任の所在」を明確にし、 事業の価値継続を最大化することです。

AI は代替案を提示できます。しかし、最終的な決定権は常に経営者に残ります。 VMS はその前提を崩さず、相談内容を Boundary、 Metrics、Decision の形に閉じます。

Monetary

資本循環と事業継続性。お金が残るか、事業として続くかを扱う。

Cultural

ドメインの歴史、美学、守るべき文化的価値を扱う。

Generative

後進へ技術・精神・判断ロジックを継承できるかを扱う。

Symptom Mapping

最初に、相談を「どの Loop の課題か」に分類します。 分類は答えではなく、どの責任境界から見直すかを決める入口です。

Physical Loop

症状: お金が残らない / 資金繰りが苦しい

Unit Economics、Throughput Accounting などを接続する。

Semantic Loop

症状: 商品が売れない / 競合に負ける

JTBD、Value Proposition Canvas などを接続する。

Value Loop

症状: 情熱が湧かない / 何のためにやっているか不明

Ikigai、Purpose-Driven Management などを接続する。

Responsibility Loop

症状: 自分がいないと回らない / 後継者が育たない

Generativity、DACI などを接続する。

最初に聞く 3 つのこと

  • current_pain: 今、一番しんどいことは何か
  • desire: どうなれば「成功」か
  • constraints: 時間・金・精神の限界値はどこか

Operation

Mode Selection

守る

Closure Mode。境界の死守と責任の閉鎖を目的にする。

攻める

Exploration Mode。構造の再発見と新しい仮説の検証を目的にする。

わからない

Diagnostic Exploration。違和感を構造化し、Closure へ橋渡しする。

初回セッション

  1. 現在の症状を入力し、どの Loop に近いかを判定する
  2. 生存境界として、金・心・美学の限界値を置く
  3. 直近の悩みに対する初回 Decision を生成する
  4. 次回レビューのために、監視する Metrics を 1 つ選ぶ

出力の最低要件

Do:
  - 具体的に実行すること
Don't:
  - 今すぐ止めること
Outcome:
  - 改善を期待するズレ

boundary_definition:
  - 今回設定・更新された境界線
metrics_to_watch:
  - 監視する Value Drift / Boundary Invasion / Responsibility Deadlock
decision_log:
  owner: 最終決定者
  status: Instant / Pending / Discard
  closure_check: この決断で閉じたループ

Domain Extensions

VMS は経営判断全般の入口です。特定領域の相談は、VMS の責任原則を保ったまま、 領域別の知識拡張として扱います。

  • VMS Marketing マーケティング相談を構造化し、比較案・不明点・回答根拠を残すためのサブセット。

Resources

VMS は、読み物ではなく AI に渡して使う仕様です。 最小読込セットを渡したうえで、相談内容を症状として入力します。

生成 AI 向けの最小読込セット

  • specs/VMS.yaml 判断原則、価値定義、責任の置き方
  • specs/prompt.txt AI への基本指示
  • specs/Input_Interface.yaml 症状の分類と不足情報の確認
  • specs/Mode_Switcher.yaml モード選択
  • specs/Output_Structure.yaml 出力フォーマット
  • specs/User_Role_Definition.yaml 利用者レベルごとの粒度調整
  • VMS GitHub Repository VMS の仕様リポジトリ。YAML とプロンプトの正本はこちらを参照してください。
